Diagnóstico del formulario actual

Qué funciona, qué está roto y qué es confuso — antes de proponer opciones

🚨 Bugs activos

  • Columna requiere_carta_aprobacion no existe en BD → 400 en cada carga Corregido ✓
  • Cuota Inicial obligatoria hardcodeada → bloquea negociaciones válidas sin cuota Corregido ✓
  • handleNext paso 2 tiene deps incompletas en useCallback → cierre estale posible en React strict
  • El campo fuentes en React Hook Form siempre es [] — el estado real vive en useFuentesPago separado, desconectado

⚠️ Deuda técnica visible

  • paso-1-info-basica.tsx y paso-1-info-basica-refactored.tsx coexisten — archivo muerto
  • Dos sistemas de validación paralelos que no se sincronizan: RHF/Zod en paso 1, estado manual en paso 2
  • valor_escritura_publica hardcodeado a $128M en el default — valor ficticio que el usuario puede omitir accidentalmente
  • Validaciones de negocio duplicadas: en useCrearNegociacion Y en useAsignarViviendaForm

✅ Lo que YA está bien

  • Modal "Ajustar plan" post-creación funciona bien — no hay que replicarlo en el wizard
  • Sistema de fuentes dinámicas desde BD (no hardcodeadas)
  • Sistema de documentos pendientes automático al crear negociación
  • Cálculo de diferencia en tiempo real (sumaCierra)
  • Sidebar con resumen financiero en tiempo real
  • Paso 3 de revisión antes de confirmar
  • Validación de documento de identidad antes de permitir negociación

El problema de UX central

El Paso 2 carga 4 fuentes de pago expandibles con muchos campos cada una — el usuario no sabe cuántas necesita ni cuáles

Lo que ve el usuario ahora en Paso 2

Cuota Inicial
Crédito Hipotecario
Subsidio Mi Casa Ya
Subsidio Caja Compensación

→ El usuario activa toggle, aparecen campos ocultos que no esperaba. Suma no cierra. Confusión.

Problema principal: "¿La suma debe ser exacta?"

El usuario activa Crédito Hipotecario e ingresa $70M.

Activa Cuota Inicial e ingresa $52M.

Total vivienda: $118.600.000

✓ $70M + $52M = $122M — Error, no cierra. Pero el usuario no sabe qué ajustar.

→ Falta un feedback claro de cuánto queda por asignar, en tiempo real, por fuente.

→ El campo "Valor escritura pública" ($128M) aparece junto a "Valor vivienda" → el usuario puede confundirse y usarlo como base.

Resumen: El wizard actual puede mantenerse como estructura (3 pasos tiene sentido) pero necesita: (1) feedback visual de distribución en vivo, (2) quitar la confusión entre valor real y valor escritura, (3) limpiar la deuda técnica interna. Hay también una alternativa de página única que puede ser más directa. Ver las opciones →

✅ Ventajas Opción A

  • Reutiliza estructura existente — menos riesgo de regresiones
  • El paso a paso guía al usuario sin overwhelm cognitivo
  • Barra de distribución en vivo → el usuario ve cuánto le falta asignar
  • Campo escritura claramente marcado como "dato legal", no afecta cálculos
  • Fuentes deshabilitadas visibles pero opacas → el usuario sabe que existen
  • Deuda técnica eliminada: un solo sistema de validación (RHF unificado)

⚠️ Compromisos

  • Requiere refactorizar el hook de fuentes para integrarlo con RHF
  • El Paso 2 sigue siendo el más complejo — no se puede simplificar mucho si los campos son obligatorios
  • 3 pasos puede sentirse largo para negociaciones simples (solo cuota + crédito)

✅ Ventajas Opción B

  • Menos clics — no hay "siguiente / atrás"
  • El usuario ve el resumen sticky siempre visible mientras llena
  • Más rápido para usuarios expertos que ya saben qué van a ingresar
  • Sin lógica de pasos → implementación más simple

❌ Desventajas Opción B

  • Puede sentirse abrumador ver todo junto — especialmente si hay muchos campos activos
  • Sin estructura de pasos, el usuario novato no sabe por dónde empezar
  • El flujo natural (primero elegir vivienda, luego configurar fuentes) se pierde
  • El botón "Crear Negociación" puede presionarse antes de completar los campos

✅ Ventajas Opción C

  • Máxima velocidad — ideal si los datos ya están claros
  • No invasivo — el modal puede abrirse desde cualquier pantalla
  • Menos código, menos superficie de bugs

❌ Desventajas Opción C

  • Los campos de entidad/radicado quedan fuera → hay que abrir "Ajustar plan" después
  • Pierde el paso de revisión antes de confirmar
  • El valor de escritura no tiene un lugar natural
  • Para negociaciones con 3-4 fuentes activas el modal se vuelve muy largo
Opinión: La Opción C funciona bien si después de crear la negociación el usuario SIEMPRE va a "Ajustar plan" para completar los detalles. Pero dado que el modal de ajuste solo está disponible para administradores, esto genera un flujo roto para asesores sin ese permiso.
